# Effort Calibration Playbook

Accumulated heuristics for choosing how much effort a task deserves: the `/effort` level, orchestration (solo vs subagents vs Workflow vs ultracode), model, review depth, and verification. Updated by the `retrospect` skill at the end of tasks.

**How to use**: before choosing effort/orchestration on a *substantial* task, skim the heuristics below and match on task-type + risk. These are evidence-based defaults, not rules — override with reason. Trivial/conversational turns don't need this.

## Default posture per lever

Think in independent levers, not one "effort" dial — a task can deserve a heavy review but a light scout.

- **Scout/understand**: default to solo targeted search. Escalate to parallel subagents only when the blast radius is genuinely unknown and wide. If the root cause is already documented (CLAUDE.md / memory / recent commits), don't re-discover it.
- **Implement**: right-size to the change. TDD for anything with logic. Avoid gold-plating and scope creep.
- **Review**: scale to risk. Self-review for isolated/low-risk; add an adversarial pass (subagent or `/code-review`) for shared code, auth/security/data, or anything hard to reverse.
- **Verify**: match the claim. Unit tests always; real-app/Playwright verification for behavior/UI changes before claiming done.

## Heuristics (dated; newest insight wins)

- **Critical / shared / auth-security-data fix → keep an adversarial review layer; keep the scout light** *(2026-07-01, Bug #695 image-auth fix)*. Adversarial review (multi-lens → verify) caught a real regression that would have shipped (empty-URL infinite-loading) plus a pre-existing bug, and changed a scope decision — clearly worth it on Critical shared viewer code. But the up-front "understand" Workflow (6 agents, ~510k tokens, 3 failed structured-output) was overkill: the root cause was already in CLAUDE.md/memory and the blast radius was findable with a few greps. Net: **review layer = worth it; understand-workflow = over-powered — scout solo instead.** A 3-4 agent review would likely have found the same top items as 12; fan-out width should scale to risk, not be maxed by default.

- **Ultracode's marginal value is the review layer, not the whole pipeline** *(2026-07-01)*. TDD + real-app verification happen regardless of effort level. When retrospecting on "was ultracode worth it", the honest question is narrower: did the *extra adversarial review* catch something real? On high-risk changes, usually yes; on isolated low-risk changes, it's ceremony. Turn the width up for risk, not for appearance.

- **Watch structured-output fan-out failure rate** *(2026-07-01)*. Explore-type subagents with complex nested output schemas failed ~50% (3/6, 3/6) in one session — wasted tokens. If a fan-out keeps failing schema validation, simplify the schema or drop to fewer agents rather than re-running the wide fan-out. *(Positive contrast, 2026-07-01: 3 plain-text Explore agents with **no output schema** succeeded 3/3 verifying review findings — dropping the schema, not the fan-out, is what fixes the failure rate.)*

- **Batch of well-specified, localized WI bugfixes → solo + TDD + one-commit-per-fix; no orchestration** *(2026-07-01, WI #693/#694/#696/#697/#700/#701)*. 6 auth-UI/logic fixes across clinic-saas were completed cleanly with zero subagents/workflows: WIs already named root cause + target files, blast radius was small, and memory carried prior context. Solo direct Read/Grep/Edit + per-fix Red→Green + per-fix commit was right-sized; spawning agents would have been pure overhead. Reserve orchestration for genuinely wide/unknown blast radius, not for a queue of small independent fixes.

- **Auth/session *timing* bugs (not reproducible in unit tests) → gather runtime evidence BEFORE fixing; verify live before claiming done** *(2026-07-01, WI #701 "reload-required login")*. The tempting guess (forceRefresh) was wrong-shaped. DevTools evidence at the failure (`oauth2/token` 200 + `/auth/me` 401×4 + tokens present in localStorage) pinpointed the real cause: `/auth/me` fired before the token was in memory, with no retry after it landed. Lesson: for post-redirect / async-provider timing, ask for network+storage state first, and since you can't self-verify a login, defer the "done" claim to a user live-test rather than shipping a plausible guess. Review-depth here is irrelevant — the safeguard is *evidence + live verification*, not an adversarial reviewer.

- **Wide fan-out review on a huge multi-subsystem PR → split angle-depth by file risk, not uniformly** *(2026-07-01, PR785 Cognito-auth 118-file review)*. A full 10-angle/9-group Workflow (86 agents, ~5.2M tokens) found real Tier-1 bugs — a migration that fails `migrate` on any DB with existing rows (empirically confirmed against the live dev DB) and an `email_verified` security bypass — that had survived the PR's own 2 prior Opus whole-branch reviews; the heavy review earned its cost on production-code groups. But the 3 test-only groups (accounts-tests/backend-integration-tests/frontend-tests, 17 of 66 findings) produced zero Tier-1/2 bugs, only duplication/coverage-gap cleanup. Rule: give the full correctness+cleanup dual-pass to production/security-critical file groups; give test-only file groups a single lighter cleanup-only pass. Sharpens the "fan-out width should scale to risk" line above with a concrete split rule.

- **Verifying a PR author's "fixed X, deferred Y" reply → solo `git show` diff + rerun the real test suite + confirm deferred items actually exist in the tracker; no orchestration needed** *(2026-07-01, PR785 follow-up)*. Reading 7 fix commits directly and running the actual backend/frontend suites (188 + 791 passed) confirmed the claims instead of trusting the reply text; querying the 8 "deferred to WI" work items confirmed they were real, assigned, and not just an excuse. Cheap, solo, and catches false-completion claims — matches "evidence before assertion."

- **Generating a review on a small/medium well-scoped PR → 3 area-split plain-text finders (backend-correctness / frontend-correctness / cleanup-altitude-conventions) + solo in-context verify of the top findings; don't expand to the code-review skill's full 8-angle × multi-vote machinery** *(2026-07-02, PR789 ~347-line/8-commit review)*. On a small PR (half tests, known blast radius) the skill nominally prescribes 8 finder angles; consolidating to 3 plain-text Explore agents (no schema, 3/3 succeeded, ~170k tokens) was right-sized and still caught 2 real Tier-2 correctness bugs the author then fixed (userInfo 4xx→503 masking a 401 re-auth; `Clinic.is_active` enforced only on the panorama surface, contradicting the PR's own "single place suffices" comment). The correctness-vs-cleanup lens separation is what surfaced both a logic bug and an efficiency bug; 8-angle fan-out would have added dedup overhead for no extra Tier-1/2. Scale finder count to diff size + risk, not to the skill's nominal ceiling. Corollary caught here: don't rubber-stamp finder claims either — one "silent-failure" finding was really pre-existing debt, not a regression, and saying so accurately mattered.

- **After resolving a merge conflict → run the affected + full test suite to catch *semantic* (not just textual) conflicts** *(2026-07-02, PR789 test_cognito_auth.py additive conflict)*. The conflict was pure additive (both sides appended tests) so textual resolution was trivial "keep both" — but the real risk was whether dev's `test_decode_unknown_kid` still held against HEAD's new negative-cache `_decode`. Running the full backend suite (196→202 passed) confirmed no semantic break before committing the merge. Textual mergeability ≠ correctness; the suite run is the actual guard, cheap and solo.

- **Schema-breaking DB cutover deploy → rehearse the migration against a restored prod dump BEFORE the maintenance window; run the deploy itself solo/interactive with read-only self-verification** *(2026-07-06, WI #690 Lightsail cutover)*. The cutover hit 3 in-window surprises — `drizzle-kit push` demanded DROP of legacy tables (runbook step 1.5's "add-only" expectation was written against an already-migrated dev DB and never tested on a true pre-migration DB), row count was 263 not the documented 34, and 3 images needed job-attribution OVERRIDES — **all three were discoverable offline from the pre-deploy dump we had already taken**. A 10-minute local rehearsal (restore dump → run push/migrate against it) would have converted mid-window firefighting into pre-window fixes. What *was* right-sized: zero agents/workflows for a guided-ops session (user executes writes on the server, agent does read-only SSH/psql verification and evidence-based diagnosis — e.g. masked-secret bytes caught by `wc -c`, migration verified by polygon-count diff). Orchestration would have added nothing; the leverage was rehearsal + per-step evidence. *(Corroborated + sharpened 2026-07-13, WI #754 prod rollout: prod SSO/IdP mutations — `az ad app update`, `cognito-idp create/update-identity-provider` — are classifier-blocked in auto mode even when the deploy itself was requested; the `!`-handoff + describe-verify division worked, and was NOT ceremony: 2 of 3 user-executed steps had first-pass defects (an invalid `openid email name` scope; step not applied at all), and one "OK" reply was still-unfixed on re-check — **read-back-verify EVERY user-executed step, never accept "done" text**. Bonus shortcut: replicating an IdP across pools is one `describe-identity-provider` on the working env — it returns the client_secret, so no Azure secret re-fetch. Front-loading all decisions in one AskUserQuestion (3 questions: slug / transfer method / PR authority) gave zero mid-flow rework.)*

- **Full-feature build from a code-complete plan (SDD) → haiku transcription-implementers + sonnet task-reviewers; concentrate the saved budget into ONE top-model whole-branch review + live verification** *(2026-07-06, WI #727 multi-clinic membership, 17 tasks / ~36 agents / ~2.5M tokens)*. When writing-plans embeds the literal code per task, haiku implementers went 9/9 first-try with zero rework; sonnet handled the 6 integration tasks. Per-task sonnet reviews caught 1 real bug (stale state on refresh failure) but returned zero findings on trivial transcription tasks (3×) and mis-rated coverage nits as Important once — task-granularity review cannot see design-level defects. The fable whole-branch review caught a spec-internal contradiction (cross-tab write-target drift — the exact accident the spec claimed to prevent) that 15 per-task reviews structurally missed, and Playwright live verification found a UX dead-end (no add-clinic entry for single-membership users) that ALL reviews missed. Levers: plan quality buys cheap implementers; cross-cutting review and real-app verification are where the irreplaceable spend goes. Consider skipping/lightening task-review on pure-transcription tasks whose full suite is green — **but only when the plan's embedded code is itself battle-tested; when the plan code is freshly authored, the transcription-task review is effectively the plan code's first review and must not be skipped** *(2026-07-09, perf-003: task review caught 2 Critical crash paths in the plan's own reference script before any live AWS round was burned)*.

- **Long interactive ops/measurement session (deploy → incident → perf campaign) → switch to orchestrator-mode BEFORE context pressure: sonnet for measurement/browser/research legwork, opus for report drafting, main context only for decisions + time-critical one-shot evidence grabs** *(2026-07-09, oral-v2 deploy + OOM incident + perf-002)*. First half ran solo (terraform, Playwright account chain ~40 tool calls, log forensics) and bloated context until the user had to demand delegation. Second half: ~10 sonnet/opus agent runs (~0.9M subagent tokens) produced all measurements/reports while main stayed workable. Two refinements proven: (a) SendMessage-resume of a measurement agent that already holds the recipes beat fresh spawns (3 follow-ups reused the sim agent); (b) the one justified inline exception was grabbing ECS `containers[].reason` before its ~1h expiry — time-critical single-query evidence stays in main. Also: live-service mutations delegated to subagents get classifier-blocked unless the user named that exact action — surface those to the user instead of delegating. *(Sharpened 2026-07-09, perf-003: terraform apply/destroy is classifier-blocked even in the MAIN session under auto mode — plan these as explicit user-run checkpoints up front (`!` prefix), and background-wrap (`(cmd > log 2>&1 &)`) any user-run command that can exceed the `!` 2-minute timeout, then watch the log. Also: known runbook pitfalls stored in memory (MSYS path mangling etc.) must be transcribed into EVERY ops subagent dispatch — one omitted pitfall cost a full measurement round.)* *(Corroborated 2026-07-10: terraform apply, ECS run-task with DB DDL, even preparing the DDL script were all blocked — conversation-level user approval does NOT unblock the classifier; hand the exact command to the user immediately instead of retrying. A user-run apply died at the 2-min `!` timeout mid-destroy and left ECS services DRAINING-stuck on stopped instances (fix: `deregister-container-instance --force`) — background-wrap from the start. Delegation split that worked: sonnet agents for build/push/endpoint-sweep (3/3 success), main for approvals and result interpretation.)* *(Contradicted in part 2026-07-10: a one-time mid-session correction does NOT stick — after 3 sonnet delegations the session regressed to full main-loop (Fable) execution for implementation, E2E loops, log forensics, docs, and commits, and the user complained a second time. The binding rule is now mechanical: choose the executor at EVERY phase boundary, and treat 3 consecutive mechanical tool calls in main as a violation → delegate the rest of the sequence. "Each next step feels small" is the failure mode, not an excuse.)*

- **Delegated report-writing (opus) → always fact-check the operational/status claims against session ground truth before delivering; prose quality is not the risk, factual drift is** *(2026-07-09, incident + perf-002 docs)*. 2 of 3 opus-written docs contained a false or stale factual claim: "テストデータ削除済み" when tenant data was actually left in place, and a recommendation section missing a constraint established later in discussion (self-bench invalid for in-window reinforcement). Spot-checking only the header/status/ops-notes/recommendation sections caught both in <5 min each. The measurement-data sections (tables passed in verbatim) were never wrong — check the sections the agent had to *infer*, not the ones it transcribed. *(Sharpened 2026-07-09, perf-003: "inferred" includes DERIVED NUMBERS — an opus report double-rounded 24.85→24.9 and propagated it to 7 places, and mis-converted MiB→GB; a top-model final review that recomputes aggregates from raw data (~60 values) caught these where prose-level spot-checks would not. Extended same day, perf-004: this applies to SOLO-authored numeric docs too — one sonnet recompute-review of a small cost estimate caught a causal mis-attribution and a citation-range drift (21.5s cited to a section that says 22s) that arithmetic self-checks structurally cannot catch, since the drift was in the citations, not the math. For numeric docs: derive every cell from one calc script, then buy exactly one independent recompute+citation review.)*

- **Design built on a benchmark's number → before implementing, enumerate the layers/conditions the bench actually traversed vs the production path; any un-traversed layer downgrades the number to a hypothesis needing one cheap production-path E2E first** *(2026-07-10, ADR-0003 inference cutover)*. perf-003's 75-78s composite skipped the ECS service scheduler (bench repeated stop/start at minute-cadence → no placement backoff; prod stops for hours) and the ET_JOB retry machinery (bench POSTed the server directly) — measured prod E2E was 236s vs SLA 120s, forcing a same-day redesign (remove ECS; docker --restart=always = the bench's own standalone lane) that then measured 50s. Corollary: unverified "X is harmless/cosmetic" assertions about third-party control-plane behavior (here: "placement failures are cosmetic") in an ADR deserve one adversarial design-review pass before deploy — this session had zero review passes on own infra changes and paid with a full rework cycle. The E2E caught everything, but paper would have been ~4h cheaper.

- **After an infra cutover, sweep ALL API endpoints, not just the changed path** *(2026-07-10, oral-v2)*. A user-demanded 29-endpoint sweep found a process-killing crash (`/job-monitor/stats` on a DB missing `ET_CLIENT_API_TOKEN` — unhandled rejection killed the whole Express process) and a schema-drift root cause (`drizzle push` vs migration-only tables), both in code untouched by the cutover. One endpoint = one process death, so partial smoke tests would have masked it as "flaky".

- **If main context already absorbed the exploration (own reads or Explore-agent reports), write the plan in main — don't spawn a separate Plan agent** *(2026-07-10)*. A Plan agent duplicating context main already held was killed mid-run; the full implementation plan was then written directly from main context in minutes. Plan agents pay off when the planning needs *new* research, not as ceremony after scouting.

- **Small live-API provisioning/ops task (≤~10 entities, contracts in one file) → main (top model) does interpretation + contract reading, then batches ALL calls into one idempotent script; that scripting IS the anti-mechanical-loop compliance** *(2026-07-10, v2 demo-user provisioning: 8 accounts/3 tenants/1 client, 0 agents, 1 script, verified by scripted logins)*. Hand-executing the ~25 API calls in main would violate the 3-consecutive-mechanical rule; one script run does not. Escalation trigger: delegate script authoring+debug to sonnet when contract discovery spans multiple files or debug exceeds ~2 cycles — at this session's scale the sonnet dispatch prompt (contracts + pitfalls transcription) would have cost as much as doing it. Also: one live probe beat more code reading (nickname lowercasing surfaced only by running).

- **Tenant/authorization-boundary worry → don't answer from reading code; probe the live endpoints multi-facet (tenantId param, X-Tenant-ID header, no-auth, and the raw file/image path) and split the result by evidence. File/image delivery routes are the likeliest leak — they sit on a different path than the JSON API and often miss the auth+tenant filter the DB layer enforces** *(2026-07-10, oral-v2 tenant-boundary probe)*. Owner suspected a broken boundary. Cross-tenant JSON probes returned SPA-HTML (origin 404 = DB layer scoped correctly), which almost ended the check as "safe" — but widening to the binary `/uploads/...` path found a route registered BEFORE `app.use(authMiddleware)` that queries by path with no `tenant_id` filter, serving another clinic's panorama X-ray to an unauthenticated caller. A code-read answer ("every query filters tenant_id") would have missed it, because the hole is a route that bypasses the query layer entirely. Probe binaries and no-auth, not just the happy JSON path.

- **Large deletion/revert to delegate → main confirms WHEN/WHY the thing was decided (ADR / commit history) BEFORE writing the delegation prompt; on return, don't trust `node --check` — grep for references to every deleted symbol (runtime ReferenceError is invisible to a syntax check) and verify any cited source (ADR/doc) actually exists** *(2026-07-10, SQS dead-code removal: sonnet 152k tok/73 tools, -252/+9)*. I mis-framed the task as "SQS was added on this branch" and my prompt instructed the agent to write an "owner rejected it today" narrative; the agent found ADR-0002 (SQS rejected 2026-07-04, code kept disabled) and corrected the story — a wrong premise becomes a wrong doc unless the agent catches it. The delegation itself was right-sized (multi-file mechanical edit), and post-receipt review earned its keep: residual-reference grep was the real guard that the DB-poller wouldn't crash at boot, and confirming ADR-0002 existed validated the doc edits. One under-spent lever: no live job E2E (no Docker) — flagged as residual risk rather than claimed done.

- **New function that parallels an existing flow → add a "matches sibling-flow conventions" review lens; generic correctness+cleanup reviews (even a top-model whole-branch) miss txn/audit/validation *consistency* defects** *(2026-07-13, WI #754 Entra auto-join, ~19 files / ~12 SDD agents)*. 6 per-task sonnet reviews + 1 opus whole-branch review ALL passed with only Minor nits, yet an external 5-lens PR review (whose angles include git-history/past-PR convention comparison) caught 2 real Tier-2 mine missed: `record_audit` placed OUTSIDE the membership `atomic()` block when every sibling flow (`invitations_accept`/`membership_disable`) keeps state+audit atomic, and an `ENABLED=true`+empty-slug silent no-op when the adjacent branch already warns on a missing clinic. Both are consistency-with-existing-code defects — invisible to correctness (behavior was fine) and cleanup (no duplication) lenses, and per-task granularity can't see them. When implementing a function modeled on an existing one, explicitly diff its txn boundary / audit / validation shape against the model. Corollary: per-task SDD reviews on this small well-specified feature returned only nits (multi-underscore test gap, stale comment); the real findings all came from whole-branch + external convention passes — **lighten per-task review on small transcription features and reinvest in one convention-comparison pass.** Contrast with 2026-07-06 WI#727 where the whole-branch review DID catch the design defect; there the defect was a spec-internal contradiction, here it was a codebase-convention mismatch — different lens, and neither generic pass covers convention-match by default.

- **Request whose premise is "X is broken/can't be done" → brainstorm the underlying goal BEFORE planning/implementing; the answer may be a different approach entirely, and front-loaded Q&A is the highest-leverage spend** *(2026-07-13, WI #754)*. "Invited users can't sign up" read as a self-registration bug; the Q&A surfaced that the real near-term need was "our own employees log in via corporate Entra ID," pivoting the whole design off #753 AdminCreateUser — which also carried a latent Cognito bug (Username=email is rejected under an email-*alias* pool, confirmed via AWS docs). Jumping to implement the literal request would have built the wrong thing AND hit the bug. Two cheap authoritative-doc lookups during scouting (alias constraint; `AllowAdminCreateUserOnly` only gates self-service `SignUp`, not federation) were load-bearing — when a design rests on a third-party service's exact behavior, one doc/probe check before implementing beats assuming.

- **Acting on a batch of code-review findings → verify each against real code before implementing; delegate the read-heavy verification by area only past ~6-8 findings; implement solo TDD, one commit per finding** *(2026-07-01, PR785 15-finding + PR789 6-finding review responses; findings were own `/code-review` output)*. 15 findings across backend+frontend: 3 area-grouped plain-text Explore agents (~118k tokens, 3/3 succeeded) confirmed all 15 accurately without polluting main context. 6 findings: verified solo in-context, no agents — right-sized. The verification is **not** ceremony: on PR789 it exposed that a prior "fix" (#702 `Clinic.is_active`) only covered the panorama surface and left `accounts` EPs bypassing, and that a 503-wrap masked a 401 re-auth case — real gaps caught by taking each reviewer claim to the code rather than blindly agreeing or blindly pushing back (the `receiving-code-review` discipline). Threshold: solo-verify up to ~6-8 findings; delegate by area (plain-text, no schema) beyond. Implementation stayed solo across all 12 fixes — consistent with the "batch of localized WI bugfixes → solo" rule above.
