# エンジニアの心得 / PR プラクティス（詳細）

共有エージェント契約（`~/.codex/AGENTS.md` ＝単一ソース、`~/.claude/CLAUDE.md` が import）の「エンジニアの心得 / PR」の全手順を退避した詳細版。
出典: Google Engineering Practices (https://google.github.io/eng-practices/) / Microsoft Code-with Engineering Playbook (https://github.com/microsoft/code-with-engineering-playbook)。

## エンジニアの心得
- コードヘルスの純増で判断する。完璧ではなく「全体のコード健全性が確実に良くなる」状態を合格ラインとし、過剰実装(gold-plating)やスコープ外の作り込みをしない
- 設計・レビューの技術判断は事実・データ・エンジニアリング原則で行い、個人の好みで決めない。スタイルはプロジェクトのスタイルガイド／既存コードの一貫性に従う（「反おもねり」がユーザーへの同調を禁じるのに対し、こちらは技術判断を好みでなく根拠で下す規律）
- リファクタリングと振る舞いの変更を同一の変更単位に混ぜない。混在するとレビューも revert も困難になる
- コードレビューはゲートキーピングではなく、品質向上・学習・コード理解の共有のための対話と捉える

## PR作成時のベストプラクティス（"自分がPRを作る側"の規律）
- PRは小さく自己完結させる。1つの目的（1機能の一部／1ユーザーストーリー）に絞り、無関係な変更を混ぜない。目安: ~100行は妥当、~1000行や50ファイル超は大きすぎる。「必要と思うより小さく」を原則とする（既存「実装前チェック: MVPスコープ」がスコープの絞り込み、こちらは1PRの粒度・レビュー可能性の規律）
- 大きなPRが避けられない場合は、事前にレビュアーへ合意を取り、変更点の説明注釈を本文に添える
- PR本文の1行目は命令形で変更内容を具体的に要約する（"Delete the RPC" 形式。"Fix bug" "Update" 等の曖昧な要約は禁止）。本文に背景・採用理由・既知の制約・関連リンク（バグ番号／設計書）を書き、将来検索される語を含める（既存「コミットメッセージは『なぜ』」をPR本文レベルに拡張したもの）
- 関連ワークアイテム／Issue をPRにリンクする（ステータス更新は CLAUDE.md「PR完了時チェック」を参照）
- PRを出す前にセルフレビューし、CI（lint／ビルド／テスト）が通ることを確認する＝ビルドを壊さない。テストは同じPRに含め、各PRはマージ後にシステムが動作する状態を保つ
- レビュー指摘には必ず応答する。修正したら resolve、対応しない場合は理由付きで「won't fix」を明示する。スコープ外の指摘は別ワークアイテム化する（スコープ内の指摘を「後でやる」で流すのは不可）
- 指摘への反論より先にコードを直す／コメントを足す（レビューツール上の弁明は将来の読み手の役に立たない）。議論はレビュースレッド上で公開で行い、長引いたら同期会話にエスカレーションする

## PRレビュールール（"自分がレビューする側"の規律）
- **PRレビューを依頼されたら、結果は必ずPRコメントとして投稿する。チャットへの出力だけで終わらせない**（投稿しないとPR作成者がレビュー結果を読めない）
- **指摘1件 = 1スレッド**で分離する（まとめ投稿は議論が紐づかず禁止）
- 各指摘は対象ファイル・行を指定したインラインコメントとして投稿する
- 各コメントには重要度ラベル（`[Tier 1: 修正必須]` / `[Tier 2: 推奨]` / `[Tier 3: cleanup]`）と「失敗シナリオ」を明記する
- レビュー後、サマリを1件だけPR全体コメント（ファイル紐付けなし）として投稿し、Tier 1件数と推奨フローを伝える
- チャット出力で終えてしまった場合は、ユーザーから指摘されたタイミングで遡って投稿し直すこと
- 投稿手段（GitHub CLI / Azure DevOps MCP 等）はプロジェクトのCLAUDE.mdで指定する
